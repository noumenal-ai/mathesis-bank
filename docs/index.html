What is this?

A public evidence bank for machine-checked learning-theory results.

Why should I care?

Every claim can be traced to an exact proof, a witness where one applies, a kernel check, and the list of axioms it depends on.

What can I do here?

Browse verified results, inspect claims, deposit new proofs, and cite immutable accessions.

What does “verified” mean?

A verified result has passed the kernel. Mathesis records the exact formal statement, the proof, a witness when applicable, and the axiom manifest. Verification does not mean the result is important, original, elegant, or aligned with informal mathematical intent. Those are separate review layers, tracked apart from the machine check.

Three kinds of evidence

Every result clears the same floor — the Lean 4 kernel accepted its proof. On top of that floor, results differ by the kind of evidence that carries them. This is a distinction of kind, not a ranking.

Proved

129 results

A general statement, established for every case by a kernel-checked proof term. Pure deduction: no instance is privileged. See one →

Witnessed

30 results

An existence statement whose concrete witness is the evidence — the kernel checks that the exhibited object has the claimed property. See one →

Mechanized-empirical

12 results

A statement about a specific executed system, whose behaviour is reflected exactly into the kernel and reasoned over — the empirical object lives inside the proof. See one →

How the label is derived: a result whose statement is about a concretely executed system (its declarations sit under an Executed namespace) is shown as mechanized-empirical; otherwise a universal statement is proved and an existential one is witnessed. It reads the gate’s own polarity field and the authors’ own namespacing — it never invents a verdict the accession does not carry.

Verify it yourself

Nothing here asks for trust. Clone the bank, build the kernel gate once, and re-derive any accession from its frozen proof export — the tool fetches the export (content-addressed, sha256-verified) and replays it through the Lean 4 kernel. It never reads the verdict the bank records about itself.

git clone https://github.com/noumenal-ai/mathesis-bank
cd mathesis-bank
bin/mathesis build                     # build the kernel gate once
bin/mathesis check MTH.R-2026-1001     # re-derive one accession
bin/mathesis check --all               # re-derive every result

Exit code is the verdict: 0 means an independent machine re-derived the proof. This is exactly what the bank’s own continuous integration runs on every change.

How it works

Each accession is immutable once written. Corrections mint a new accession; they never edit the old one.

  1. Define terms. Definitions enter the Dictionary.
  2. Freeze a claim. A claim can be recorded before any proof is attempted (optional).
  3. Deposit a result. A proof attempts to discharge the claim.
  4. Check and accession. The kernel re-derives the result and Mathesis mints an immutable accession.
  5. Correct by adding, not editing. Errors mint new accessions; old records stay visible.
